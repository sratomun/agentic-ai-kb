---

Agents are only as effective as the tools we give them. We share how to write high-quality tools and evaluations, and how you can boost performance by using Claude to optimize its tools for itself.

The Model Context Protocol (MCP) can empower LLM agents with potentially hundreds of tools to solve real-world tasks. But how do we make those tools maximally effective?

This post covers: building/testing prototypes of tools; creating and running comprehensive evaluations with agents; collaborating with agents like Claude Code to automatically increase tool performance. It concludes with key principles: choosing the right tools to implement (and not to); namespacing tools; returning meaningful context; optimizing for token efficiency; prompt-engineering tool descriptions.

## What is a tool?

Deterministic systems produce the same output every time given identical inputs; non-deterministic systems—like agents—can generate varied responses even with the same starting conditions. Traditional software establishes a contract between deterministic systems (e.g., `getWeather("NYC")` always fetches the same way). Tools are a new kind of software which reflects a contract between deterministic systems and non-deterministic agents. When a user asks "Should I bring an umbrella today?", an agent might call the weather tool, answer from general knowledge, or ask a clarifying question. This means rethinking our approach: instead of writing tools and MCP servers the way we'd write functions and APIs for other developers, we need to design them for agents.

## How to write tools

### Building a prototype
Stand up a quick prototype and test locally. If using Claude Code to write tools, give Claude documentation for any libraries/APIs/SDKs (LLM-friendly docs are often in flat `llms.txt` files). Wrap tools in a local MCP server or Desktop extension (DXT) to connect and test in Claude Code or Claude Desktop. Connect via `claude mcp add <name> <command> [args...]`. Tools can also be passed directly into Anthropic API calls for programmatic testing.

### Running an evaluation
Measure how well Claude uses your tools. Generate lots of evaluation tasks grounded in real-world uses, with realistic data sources.

Strong tasks (require multiple, potentially dozens of tool calls):
- Schedule a meeting with Jane next week to discuss our latest Acme Corp project. Attach the notes from our last project planning meeting and reserve a conference room.
- Customer ID 9182 reported they were charged three times for a single purchase attempt. Find all relevant log entries and determine if any other customers were affected.
- Customer Sarah Chen just submitted a cancellation request. Prepare a retention offer: (1) why they're leaving, (2) what offer would be most compelling, (3) any risk factors.

Weaker tasks: "Schedule a meeting with jane@acme.corp next week." / "Search the payment logs for purchase_complete and customer_id=9182."

Each prompt should be paired with a verifiable response/outcome (exact string comparison, or Claude-as-judge). Avoid overly strict verifiers that reject correct responses for formatting differences.

Run evaluations programmatically with direct LLM API calls using simple agentic loops (while-loops wrapping alternating LLM API and tool calls), one loop per task. Instruct agents to output reasoning and feedback blocks (triggers chain-of-thought). Collect metrics: top-level accuracy, runtime, number of tool calls, token consumption, tool errors.

**Analyzing results:** What agents omit can be more important than what they include. LLMs don't always say what they mean. Observe where agents get confused; review raw transcripts. Lots of redundant tool calls suggest pagination/token-limit rightsizing; lots of errors for invalid parameters suggest clearer descriptions/examples. When launching Claude's web search tool, identified that Claude was needlessly appending `2025` to the query parameter, biasing results (fixed via tool description).

### Collaborating with agents
Concatenate transcripts from evaluation agents and paste into Claude Code. Claude is expert at analyzing transcripts and refactoring many tools at once. Most advice in this post came from repeatedly optimizing internal tool implementations with Claude Code. Used held-out test sets to avoid overfitting — these revealed performance improvements beyond even "expert" tool implementations, whether manually written by researchers or generated by Claude itself.

## Principles for writing effective tools

### Choosing the right tools for agents
More tools don't always lead to better outcomes. A common error: tools that merely wrap existing software functionality or API endpoints. LLM agents have limited context, whereas computer memory is cheap. A tool returning ALL contacts wastes context. Build a few thoughtful tools targeting specific high-impact workflows. Examples of consolidation:
- Instead of `list_users`, `list_events`, `create_event` → implement `schedule_event` which finds availability and schedules.
- Instead of `read_logs` → `search_logs` returning only relevant lines with context.
- Instead of `get_customer_by_id`, `list_transactions`, `list_notes` → `get_customer_context` compiling all recent/relevant info at once.

Too many overlapping tools distract agents from efficient strategies.

### Namespacing your tools
Namespacing (grouping related tools under common prefixes) helps delineate boundaries: by service (`asana_search`, `jira_search`) and by resource (`asana_projects_search`, `asana_users_search`). Prefix- vs suffix-based namespacing had non-trivial effects on tool-use evaluations; effects vary by LLM.

### Returning meaningful context from your tools
Return only high-signal information. Prioritize contextual relevance over flexibility; eschew low-level technical identifiers (`uuid`, `256px_image_url`, `mime_type`). Resolving arbitrary alphanumeric UUIDs to semantically meaningful language (or a 0-indexed scheme) significantly improves Claude's precision and reduces hallucinations. Expose a `response_format` enum (`concise` / `detailed`) so the agent controls verbosity. Response structure (XML, JSON, Markdown) affects performance—no one-size-fits-all; LLMs perform better with formats matching training data.

### Optimizing tool responses for token efficiency
Implement pagination, range selection, filtering, and/or truncation with sensible defaults. For Claude Code, tool responses are restricted to 25,000 tokens by default. Prompt-engineer error responses to communicate specific, actionable improvements rather than opaque error codes.

### Prompt-engineering your tool descriptions
One of the most effective methods. Think of how you'd describe your tool to a new hire. Make implicit context explicit. Input parameters should be unambiguously named (`user_id`, not `user`). Even small refinements yield dramatic improvements: Claude Sonnet 3.5 achieved state-of-the-art performance on SWE-bench Verified after precise refinements to tool descriptions, dramatically reducing error rates.

## Looking ahead
We need to re-orient software development from predictable deterministic patterns to non-deterministic ones. Effective tools are intentionally and clearly defined, use agent context judiciously, can be combined in diverse workflows, and enable agents to intuitively solve real-world tasks. With a systematic, evaluation-driven approach, as agents become more capable the tools they use will evolve alongside them.
