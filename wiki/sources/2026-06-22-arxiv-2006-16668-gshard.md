# GShard: Scaling Giant Models with Conditional Computation and Automatic Sharding

**arXiv [2006.16668](https://arxiv.org/abs/2006.16668)** · 2020-06-30 · Dmitry Lepikhin, HyoukJoong Lee, Yuanzhong Xu, Dehao Chen, Orhan Firat, Yanping Huang, Maxim Krikun, Noam Shazeer, Zhifeng Chen

**Significance.** The paper that made sparsely-gated Mixture-of-Experts practical at scale — a 600B-parameter MoE Transformer trained in 4 days, with the automatic-sharding (GShard/XLA SPMD) machinery that seeded modern trillion-parameter MoE training.

## Abstract
Neural network scaling has been critical for improving the model quality in many real-world machine learning applications with vast amounts of training data and compute. Although this trend of scaling is affirmed to be a sure-fire approach for better model quality, there are challenges on the path such as the computation cost, ease of programming, and efficient implementation on parallel devices. GShard is a module composed of a set of lightweight annotation APIs and an extension to the XLA compiler. It provides an elegant way to express a wide range of parallel computation patterns with minimal changes to the existing model code. GShard enabled us to scale up multilingual neural machine translation Transformer model with Sparsely-Gated Mixture-of-Experts beyond 600 billion parameters using automatic sharding. We demonstrate that such a giant model can efficienctly be trained on 2048 TPU v3 accelerators in 4 days to achieve far superior quality for translation from 100 languages to English compared to the prior art.

## From the paper (full-text)
**Contribution / method.** A sparsely-scaled Transformer for massively multilingual NMT (the "M4" task, 100 languages → English). Every *other* feed-forward (FFN) sublayer in both encoder and decoder is replaced by a Position-wise Mixture-of-Experts layer: E parallel FFN experts plus a softmax gating network using a variant of **top-2 gating**. Each token is routed to at most two experts; outputs are a gate-weighted average. Load balance is enforced by (a) an **expert capacity** threshold ≈ O(N/E) per expert with overflowed tokens passed through via residual connections, (b) **local group dispatching** (tokens split into G groups, each with fractional per-expert capacity 2N/(G·E)), (c) a differentiable **auxiliary load-balancing loss** ℓ_aux added as L = ℓ_nll + k·ℓ_aux (approximating mean-square of per-expert routed fraction using mean gates m_e), and (d) **random routing** to the 2nd-best expert with probability ∝ its gate weight. The whole forward MoE layer is expressed in a handful of einsums. The systems contribution is **GShard**: lightweight per-tensor sharding annotations (`replicate`, `split`, `shard`) plus an **XLA SPMD partitioner** that compiles a single program for all devices (so compilation time is ~constant regardless of partition count), inserts collective comms (AllReduce, AllToAll, AllGather, CollectivePermute), and handles uneven partitioning via halo exchange. Experts are sharded across devices while non-MoE layers are replicated.

**Results.** The largest model, **MoE(2048E, 36L) = 600B parameters** (2048 experts per MoE layer, 36 enc+dec layers), was trained on **2048 TPU v3 cores in 4.0 days** (22.4 TPU v3 core-years), reaching **44.3 average BLEU / +13.5 ΔBLEU** over 100 bilingual baselines. The dataset is a web-mined corpus covering **100 languages** to/from English (~25B examples total; ~13B used for the to-English direction). Quality scaled with size across the model family: MoE(512E,36L)=150B→43.7 BLEU, MoE(2048E,12L)=200B→41.3, MoE(128E,36L)=37B→39.0, MoE(128E,12L)=12.5B→36.7. The strongest dense baseline, T(96L)=2.3B, hit only 36.9 BLEU (+6.1 ΔBLEU) yet needed ~235.5 TPU v3 core-years (~10× more compute, ~6 weeks). Training all 100 bilingual baselines would have cost 29 TPU v3 core-years vs 22 for the 600B MoE. A 1-trillion-parameter MoE(2048E, 60L) was run for scalability but hit numerical-stability issues and was excluded from quality results. Systems efficiency: step time grows only **1.7×** when scaling the model 16× (128→2048 experts); per-device memory is roughly O(1) in number of experts; AllToAll cost scales ~O(√D); matmuls hit >85% peak FLOPS.

**Takeaway.** Conditional computation (sparse MoE) plus compiler-level automatic sharding turns "bigger" from a super-linear cost problem into a sub-linear one — better quality *and* an order-of-magnitude cheaper than dense scaling for the same hardware.
